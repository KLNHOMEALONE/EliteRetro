## 4. New Critical Issues (Not in Original Review)

### Exploits / Data Integrity
**NE-1: MarketSystem Negative Price Exploit** (MarketSystem.cs:130)
- Severity: HIGH
- Price formula allows negative prices when `BasePrice + (rng) + ecoIndex * EconomyPriceFactor < 0`
- Example: Food at economy 7: `20 + (0-31) + 7*(-3) = -1` → -4 credits per unit
- Player gains money by buying negatively-priced commodities
- Fix: Clamp price to minimum 1 before `* 4`, or use `Math.Max(1, price)`

**NE-2: MarketSystem Availability Wrap-Around** (MarketSystem.cs:146)
- Severity: HIGH
- Availability uses bitwise AND (`& 63`) on signed intermediate values
- Negative values wrap to high availability (e.g., -1 & 63 = 63)
- System with high economy and low base quantity shows 63 tons of everything
- Fix: Use `((qty % 64) + 64) % 64` for proper unsigned modulo, or cast to byte before AND

**NE-3: TaskScheduler Exception Propagation** (TaskScheduler.cs:46-55)
- Severity: HIGH
- No try/catch around registered task actions
- One failing task (null reference in AI) stops all subsequent tasks for the frame
- Can leave systems in inconsistent state
- Fix: Wrap each task in try/catch, log error, continue processing

**NE-4: Buy/Sell Methods Don't Validate Negative Prices** (MarketSystem.cs:155-191)
- Severity: HIGH (in combination with NE-1)
- If Prices[idx] is negative, Buy adds credits (exploit), Sell subtracts credits (penalty)
- Fix: Validate prices on load (clamp to [1, 255]) or at runtime

### Resource Leaks (Memory/GPU)
**NE-5: WhitePixel Texture Leaks in All Renderers** (9 files)
- Severity: HIGH (cumulative)
- Every renderer (Wireframe, Circle, Ellipse, Sun, Explosion, Stardust, Ring, Scanner, Hud) creates `Texture2D _whitePixel`
- None implement IDisposable
- Scene transitions leak GPU resources
- Fix: Create shared static WhitePixel singleton, or implement IDisposable on all renderers and dispose in Scene.UnloadContent

**NE-6: MainMenuScene/ SpaceScene Texture Leaks** (MM-1, SP-3)
- Severity: MEDIUM
- `MainMenuScene._whitePixel`, `SpaceScene._pixelTexture` never disposed
- Also SpaceScene: `_circleRenderer`, `_ringRenderer`, `_sunRenderer` created but never used (dead allocations)
- Fix: Dispose in UnloadContent(); remove unused fields

**NE-7: Explosion/Stardust/Sun Per-Frame `Random` Allocations** (ExplosionRenderer.cs:130, SunRenderer.cs:88)
- Severity: MEDIUM (GC pressure)
- New Random() instantiated every frame in Draw methods
- Should cache Random instance or pre-compute fringe data
- Also ExplosionRenderer.DrawCloud uses new Random() per frame

**NE-8: WireframeRenderer Per-Frame Allocations** (WireframeRenderer.cs:199-201, 243, 156-164)
- Severity: MEDIUM-HIGH
- Allocates `List<int>[] edgeFaces` (N lists) and `HashSet<int>[] faceVertexSets` (N sets) every frame
- O(E×V×F) edge-face matching is computationally expensive
- Per-vertex and per-edge projection arrays allocated every frame
- Fix: Object pool for collections, cache projections, use indexed for-loops instead of LINQ

### Correctness / Gameplay Bugs
**NE-9: CheckPlanetCrash Scale Factor Wrong** (CollisionSystem.cs:296)
- Severity: HIGH
- Uses hardcoded `0.0001f` scale factor on PlanetRadius (24576)
- Result: crash radius = 2.45 units (tiny)
- Entities spawn 100-300 units away → crashes never trigger
- Comment suggests this converts to "local coords" but inconsistent with FlightScene RenderScale = 0.001f
- Fix: Use consistent coordinate system; likely should compare in Elite internal units without scale, or use correct simulation-space radius

**NE-10: DockingSystem Approach Angle Check Inverted** (DockingSystem.cs:86)
- Severity: MEDIUM-HIGH
- `CheckApproachAngle` returns true when `fixedDot <= 214` (dot product < 0.84)
- This means angle > 33° passes check — backwards!
- Should be `fixedDot >= ApproachAngleThreshold` to require facing the station
- Likely prevents docking even when properly aligned

**NE-11: DockingSystem Safe Cone Check Uses Wrong Coordinate Space** (DockingSystem.cs:104-113)
- Severity: MEDIUM
- `CheckSafeCone` transforms station→ship vector into station space
- But original Elite checks if ship is behind station (in docking slot)
- Current check `localDir.Z >= 0.35f` may not match intended cone geometry
- Could reject valid docking approaches

**NE-12: ShipAISystem Type Confusion (ShipClass vs NewbFlags)** (ShipAISystem.cs:107, 263)
- Severity: MEDIUM
- Casts `ship.Blueprint.ShipClass` (byte) to `NewbFlags` enum
- ShipClass values: 0=Innocent, 1=BountyHunter, 2=Pirate, 3=Cop
- NewbFlags values: Trader=1, BountyHunter=2, Hostile=4, Pirate=8, Cop=64
- Pirate (ShipClass=2) becomes NewbFlags.BountyHunter (value 2)
- AI hostility logic uses wrong flags → incorrect behavior
- Also `ShipBlueprint.Personality` property is dead code (never used)

**NE-13: StardustRenderer Off-Screen Check Wrong** (StardustRenderer.cs:158)
- Severity: MEDIUM
- Checks `screenX > center.X + 1024 + 10` but center.X = 512
- Right edge is at 512+512=1024, not 512+1024=1536
- Stars beyond actual screen are still drawn (waste)
- Same for Y: uses 768 but screen height is 384 from center
- Fix: Should be `center.X + 512 + 10` and `center.Y + 384 + 10`

**NE-14: StardustRenderer Center Stars NaN on Normalize** (StardustRenderer.cs:183-184)
- Severity: LOW
- When star at exact screen center, dir = (0,0), Normalize → NaN
- Check `dir.Length() < 0.01f` catches this, sets dir = UnitX
- Results in arbitrary dash direction for center stars

**NE-15: RingRenderer Duplicate Methods** (RingRenderer.cs:201-273)
- Severity: LOW (maintenance burden)
- `DrawAxisAlignedRings` is near-duplicate of `DrawRings`
- Only difference: one applies tilt rotation, one doesn't
- Should refactor into single method with parameter
